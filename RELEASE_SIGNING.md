# Release Signing Guide

This document describes how to sign releases for this project to verify authenticity and integrity.

## Why Sign Releases?

Signed releases allow users to verify that:
- The release was created by a trusted maintainer
- The artifacts haven't been tampered with since signing
- The SBOM and other release assets are authentic

## Prerequisites

1. **GPG Key**: You need a GPG key pair. If you don't have one:
   ```bash
   # Generate a new GPG key
   gpg --full-generate-key
   
   # List your keys to get the key ID
   gpg --list-secret-keys --keyid-format LONG
   ```

2. **GitHub Configuration**: Add your GPG public key to your GitHub account:
   - Go to Settings → SSH and GPG keys → New GPG key
   - Export your public key: `gpg --armor --export your-email@example.com`
   - Paste the output into GitHub

## Signing a Release

### Step 1: Create a Release Tag

```bash
# Create and sign the tag
git tag -s v1.0.0 -m "Release v1.0.0"

# Push the signed tag
git push origin v1.0.0
```

### Step 2: Prepare Release Artifacts

```bash
# Create source archive
git archive -o claude-light-v1.0.0.tar.gz --prefix=claude-light-v1.0.0/ v1.0.0

# Generate SBOM (if not already generated by CI)
cyclonedx-py requirements -i requirements.txt -o sbom.json
```

### Step 3: Sign the Artifacts

```bash
# Sign the tarball
gpg --armor --detach-sign claude-light-v1.0.0.tar.gz

# Sign the SBOM
gpg --armor --detach-sign sbom.json

# Create checksums
sha256sum claude-light-v1.0.0.tar.gz sbom.json > SHA256SUMS

# Sign the checksums
gpg --armor --detach-sign SHA256SUMS
```

### Step 4: Upload to GitHub Releases

1. Go to GitHub → Releases → Create a new release
2. Select the tag you created
3. Upload all files:
   - `claude-light-v1.0.0.tar.gz` (source archive)
   - `claude-light-v1.0.0.tar.gz.asc` (signature)
   - `sbom.json` (Software Bill of Materials)
   - `sbom.json.asc` (SBOM signature)
   - `SHA256SUMS` (checksums)
   - `SHA256SUMS.asc` (checksum signature)

## Verifying a Release

Users can verify the release authenticity:

```bash
# Import the maintainer's GPG key
gpg --recv-keys <KEY_ID>

# Verify the signature
gpg --verify claude-light-v1.0.0.tar.gz.asc

# Verify checksums
sha256sum -c SHA256SUMS
```

## Automated Release Signing (GitHub Actions)

For automated signing, create a workflow `.github/workflows/release.yml`:

```yaml
name: Create Release

on:
  push:
    tags:
      - 'v*'

permissions:
  contents: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      
      - name: Generate SBOM
        run: |
          pip install cyclonedx-bom
          cyclonedx-py requirements -i requirements.txt -o sbom.json
      
      - name: Create source archive
        run: |
          git archive -o claude-light-${{ github.ref_name }}.tar.gz \
            --prefix=claude-light-${{ github.ref_name }}/ ${{ github.ref_name }}
      
      - name: Create Release
        uses: softprops/action-gh-release@v2
        with:
          files: |
            claude-light-${{ github.ref_name }}.tar.gz
            sbom.json
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## Best Practices

- **Key Security**: Keep your private key secure, preferably on a hardware token (YubiKey)
- **Key Expiry**: Set an expiration date and renew periodically
- **Multiple Signatures**: For critical releases, consider having multiple maintainers sign
- **Transparency**: Publish your key fingerprint in README.md or SECURITY.md
